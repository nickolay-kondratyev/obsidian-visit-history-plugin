# Task 1.1: Create VaultNode Type

> **Phase:** 1 — Core Data Structures
> **Depends on:** Task 0.1
> **Blocks:** All view component tasks

## Context

The prototype defines this data contract inline as a comment. We need it as a proper TypeScript interface in `src/core/data/`.

## What to Create

**File:** `src/core/data/VaultNode.ts`

```typescript
/**
 * Tree node for the vault heatmap visualization.
 *
 * Folder nodes have `children`, no file-specific fields.
 * Leaf nodes have `type`, `size`, and timestamp fields, no `children`.
 *
 * This is a DUAL-PURPOSE type used by both:
 * - The treemap data adapter (builds the tree from vault files)
 * - The view components (render the tree as SVG)
 *
 * @see FileTimeMetadata for the per-file timestamp tracking used by
 *      the visit history system independently of this tree structure.
 */
export interface VaultNode {
  /** Display name — file name (leaf) or folder name (branch). */
  name: string;

  /** Present only on folder nodes. Absent on leaf nodes. */
  children?: VaultNode[];

  // ── Leaf-only fields (undefined on folders) ──────────────────

  /** File extension category. Only defined for leaf (file) nodes. */
  type?: 'md' | 'canvas' | 'excalidraw';

  /** File size in bytes. Only defined for leaf nodes. */
  size?: number;

  /** Unix milliseconds — from TFile.stat.ctime */
  createdAt?: number;

  /** Unix milliseconds — from TFile.stat.mtime */
  lastModifiedAt?: number;

  /**
   * Unix milliseconds — plugin-tracked last open time.
   * `null` means the file has never been opened (or tracking not yet active).
   * `undefined` on folder nodes.
   */
  lastVisitedAt?: number | null;
}
```

## Relationship to FileTimeMetadata

- `FileTimeMetadata` (existing, `src/core/data/FileTimeMetadata.ts`) is the **flat per-file tracking record**: `{ createdMs, modifiedMs, visitedMs }`.
- `VaultNode` is the **hierarchical visualization node** with optional `lastVisitedAt` (among other fields).
- `buildVaultTree` (in `src/viewModel/`) merges `FileTimeMetadata.visitedMs` → `VaultNode.lastVisitedAt`.
- They remain **separate types** — different purposes, different consumers.

## Notes

- This intentionally does NOT import from `'obsidian'` — it's a pure data type usable by core, viewModel, and view.
- Field naming uses camelCase `lastVisitedAt` (matching prototype's data contract and JS conventions), while `FileTimeMetadata` uses `visitedMs` (existing codebase convention). The adapter in `buildVaultTree` handles the mapping.
- The `children?` / `type?` / `size?` optional pattern means folder nodes have `children` but no leaf fields, and leaf nodes have leaf fields but no `children`. This is a discriminated union in practice, though not enforced by the type system.
